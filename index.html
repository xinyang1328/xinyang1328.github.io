属性 作用
navigationStart 在同一个浏览器上下文中，前一个网页（与当前页面不一定同域）unload 的时间戳，如果无前一个网页 unload ，则与 fetchStart 值相等
unloadEventStart 前一个网页（与当前页面同域）unload 的时间戳，如果无前一个网页 unload 或者前一个网页与当前页面不同域，则值为 0
unloadEventEnd 和 unloadEventStart 相对应，返回前一个网页 unload 事件绑定的回调函数执行完毕的时间戳

domainLookupEnd DNS 域名查询完成的时间，如果使用了本地缓存（即无 DNS 查询）或持久连接，则与 fetchStart 值相等
connectStart:HTTP（TCP） 开始建立连接的时间，如果是持久连接，则与 fetchStart 值相等
connectEnd:HTTP（TCP） 完成建立连接的时间（完成握手），如果是持久连接，则与 fetchStart 值相等。
secureConnectionStart HTTPS 连接开始的时间，如果不是安全连接，则值为 0
requestStart HTTP 请求读取真实文档开始的时间（完成建立连接），包括从本地读取缓存（连接错误重连时，这里显示的也是新建立连接的时间）
responseStart HTTP 开始接收响应的时间（获取到第一个字节），包括从本地读取缓存
responseEnd HTTP 响应全部接收完成的时间（获取到最后一个字节），包括从本地读取缓存
domLoading 开始解析渲染 DOM 树的时间，此时 Document.readyState 变为 loading，并将抛出 readystatechange 相关事件
domInteractive 完成解析 DOM 树的时间，Document.readyState 变为 interactive，并将抛出 readystatechange 相关事件
domContentLoadedEventStart DOM 解析完成后，网页内资源加载开始的时间，在 DOMContentLoaded 事件抛出前发生
domContentLoadedEventEnd DOM 解析完成后，网页内资源加载完成的时间（如 JS 脚本加载执行完毕）
domComplete DOM 树解析完成，且资源也准备就绪的时间，Document.readyState 变为 complete，并将抛出 readystatechange 相关事件
loadEventStart load 事件发送给文档，也即 load 回调函数开始执行的时间
loadEventEnd load 事件的回调函数执行完毕的时间

注：

connectEnd：如果在传输层发生了错误且重新建立连接，则这里显示的是新建立的连接完成的时间；这里握手结束，包括安全连接建立完成、SOCKS 授权通过

connectStart：如果在传输层发生了错误且重新建立连接，则这里显示的是新建立的连接开始的时间

domInteractive：只是 DOM 树解析完成，这时候并没有开始加载网页内的资源

loadEventStart：如果没有绑定 load 事件，值为 0

跨域

前言

产生原因：

浏览器同源政策：如果两个页面的协议、域名、端口都相同，那么就可以说明这两个页面有同样的源。这两个页面就可以相互访问，反之，则在没有采取措施的情况下，是无法访问的

在前端开发者中，很容易遇到跨域问题，比如在a域获取了一些信息，需要发送给b域，那么，如果没有进行跨域解决方案，那么就会失败

CORS

前言

CORS：跨域资源共享，使用额外的http头来告诉浏览器，让运行在一个origin上的web应用被准许访问来自不同原服务器上指定的资源，当一个资源从与该资源本身所在的服务器不同的域、协议或端口请求一个资源时，资源会发起一个跨域http请求

需要浏览器和服务器同时支持，才可以实现。当前几乎所有浏览器都支持CORS，IE都不能低于10，CORS的整个过程都是由浏览器自动完成。

功能概述：CROS标准新增了一组http首部字段，允许服务器声明哪些源站通过浏览器有权限访问哪些资源；对那些可能对服务器数据产生副作用的http请求方法(get以外的请求，或者搭配某些mime类型的post请求)，浏览器必须首先使用options方法发起一个预检请求，从而获知服务端是否允许跨域请求

http学习（1）

发表于 2020-02-21 | 分类于 前端

概述

web客户端与服务器是如何通信的

1、客户端 —-request请求—–》服务器

2、服务器接收到请求—-转发—-》应用处理和解析

3、服务器解析完毕—-返回response(业务需要返回的结果)—》客户端

4、客户端解析response，展示结果

web内容资源来源

web内容都是存储在web服务器上的，客户端发出请求，服务器返回数据

web事务如何工作

事务：载体为http报文，即由请求报文和响应报文组成

请求报文：包含请求指令，常用请求指令是：get、post

响应报文：包含响应状态码

http历史

http/0.9

最初的版本，适用于各种数据信息的简洁快速，但是远不能满足日益发展的各种应用的需要。是一个交换信息的无序协议，仅仅限于文字。

1、只有一个命令get

2、没有header等描述数据的信息

3、服务器发送完毕后，就关闭tcp连接

http/1.0

成为最重要的面向事务的应用层协议，对每次请求/响应建立并拆除一个连接。简单，易于管理

1、增加了很多命令

2、增加响应码和header

3、多字符集支持、多部分发送、权限、缓存等

http/1.1

规定了连接方式及类型

1、持久连接

2、pipeline

3、增加host和其他一下命令

http/2.0

1、所有数据以二进制传输

2、在一个连接中发送多个请求可不按照顺序执行

3、头信息压缩以及推送等提高效率的功能

英特网上安装的大量http架构组件中的一部分

1、代理服务器：位于服务器和客户端直接的http实体

2、缓存服务器：http仓库，使常用网页的副本
可以保存在离客户端更近的地方

3、网关：连接与其他应用程序的特色web服务器

4、agent代理：发起自动http请求的半智能web客户端

请求过程

1、客户端发起http请求

2、向dns请求解析

3、dns返回解析的ip地址

4、向协议(http、ftp等)请求资源（应用层

5、应用层交给tcp协议(将http报文分割成多个报文段，且每个报文段可靠的传给对方)（运输层

6、交给ip协议(搜索对象地址，边中转边传输二进制数据)（网络层

7、经过路由器，ip协议(接收到二进制数据)将数据传给tcp协议

8、tcp协议(重组接收到的报文段)将报文发送给http协议

9、tcp协议(将请求的内容封装处理)传送给服务器

10、服务器(进行逻辑处理，并将数据进行回传)将数据传送给客户端

http报文

包含内容

起始行：保温第一行，在请求报文中说明要做些撒子，在响应报文中说明出现了啥子情况

首部字段：起始行后面的就是首部字段（有零行或者多行）。每个首部字段都包含一个名字和一个值，中间用：隔开，有点像键值对。首部以一个空行结束

主体：空行之后的，可选滴，包含所有类型的数据。请求主体中包括了要发送给web的数据，响应主体包含的是要返回给客户端的数据。

起始行和首部都是文本形式且都是结构化的，而主体可以包含各种数据

http Code

定义服务器对请求的处理结果，各个区间的code有各自的语义，最好能通过code判断结果

100~199：信息

消息 描述
100 服务器仅接收到部分请求，但是一旦服务器并没有拒绝该请求，客户端应该继续发送其余的请求。
101 服务器转换协议：服务器将遵从客户的请求转换到另外一种协议。

200~299:成功

消息 描述
200 请求成功（其后是对GET和POST请求的应答文档。）
201 请求被创建完成，同时新的资源被创建。
202 供处理的请求已被接受，但是处理未完成。
203 文档已经正常地返回，但一些应答头可能不正确，因为使用的是文档的拷贝。
204 没有新文档。浏览器应该继续显示原来的文档。如果用户定期地刷新页面，而Servlet可以确定用户文档足够新，这个状态代码是很有用的。
205 没有新文档。但浏览器应该重置它所显示的内容。用来强制浏览器清除表单输入内容。
206 客户发送了一个带有Range头的GET请求，服务器完成了它。

300~399：重定向

消息 描述
300 多重选择。链接列表。用户可以选择某链接到达目的地。最多允许五个地址。
301 所请求的页面已经转移至新的url。
302 所请求的页面已经临时转移至新的url。
303 所请求的页面可在别的url下被找到。
304 未按预期修改文档。客户端有缓冲的文档并发出了一个条件性的请求（一般是提供If-Modified-Since头表示客户只想比指定日期更新的文档）。服务器告诉客户，原来缓冲的文档还可以继续使用。
305 客户请求的文档应该通过Location头所指明的代理服务器提取。
306 此代码被用于前一版本。目前已不再使用，但是代码依然被保留。
307 被请求的页面已经临时移至新的url。

400~499：客户端错误

消息 描述
400 服务器未能理解请求。
401 被请求的页面需要用户名和密码。
402 此代码尚无法使用。
403 对被请求页面的访问被禁止。
404 服务器无法找到被请求的页面。
405 请求中指定的方法不被允许。
406 服务器生成的响应无法被客户端所接受。
407 用户必须首先使用代理服务器进行验证，这样请求才会被处理。
408 请求超出了服务器的等待时间。
409 由于冲突，请求无法被完成。
410 被请求的页面不可用。
411 “Content-Length” 未被定义。如果无此内容，服务器不会接受请求。
412 请求中的前提条件被服务器评估为失败。
413 由于所请求的实体的太大，服务器不会接受请求。
414 由于url太长，服务器不会接受请求。当post请求被转换为带有很长的查询信息的get请求时，就会发生这种情况。
415 由于媒介类型不被支持，服务器不会接受请求。
416 服务器不能满足客户在请求中指定的Range头。
417 执行失败。
418 锁定的错误。

500~599：服务器错误

消息 描述
500 请求未完成。服务器遇到不可预知的情况。

cookie

释义：

Cookie，有时也用其复数形式 Cookies，指某些网站为了辨别用户身份、进行 session 跟踪而储存在用户本地终端上的数据，也可以叫为浏览器缓存

作用：

访问一个网站后，在本地存储一些关于介个网站的信息，下次再访问的时候就阔以减少一些步骤，从而提高速度。

比如：你有一次登录了一个网站，输入您的姓名和密码后，然后发现第二次，咦~~这个网站咋自己登录了嘞？？？

莫要方丈，这就素cookie的作用。

内容：

主要内容包括：名字，值，过期时间，路径及域。

存储一般按 key - value 的形式存放滴。

过期时间是可以自己设置的，如果木有设置，那么关了浏览器后就消失了，并且其被存于内存之中；如果设置了过期时间，那么到了过期时间才会被消除，存储地点为硬盘。

路径和域其实就是把网站的url分开了，比如介个：

不足：

虽然cookie很有用、留批，但素，人无完人，cookie无完cookie，是吧，cookie也是有大大滴不足der。

它呢，因为是把你的隐私呀，全部放在硬盘里的，所以这就大大加深了被盗取用于一些不可描述地方的可能性。所以嘞，通过病毒盗取浏览器的cookie，直接通过非法手段得来的cookie骗取网站的信任，就会导致个人信息泄露。

所以，cookie有风险，用时需谨慎（但素完全不用也是不太可能滴….

session

释义：

在计算机中，尤其是在网络应用中，称为“会话控制”。Session对象存储特定用户会话所需的属性及配置信息。这样，当用户在应用程序的Web页之间跳转时，存储在Session对象中的变量将不会丢失，而是在整个用户会话中一直存在下去。当用户请求来自应用程序的 Web页时，如果该用户还没有会话，则Web服务器将自动创建一个 Session对象。当会话过期或被放弃后，服务器将终止该会话。Session 对象最常见的一个用法就是存储用户的首选项。例如，如果用户指明不喜欢查看图形，就可以将该信息存储在Session对象中。

以上，取自百度百科（就素不想一个一个敲了 略略略

下面的嘞，就素我看他们的博客，然后按自己理解的意思敲的：

session是在服务器中存放数据的（废话…），结构类似于Object，由键值对组成的。

当浏览器发送第一次请求的时候，服务器自动生成了一个session_id和一个存放session的类似于一个表一样的东东，就相当于为这个session开辟了一个空间，然后通过响应把这个发到浏览器。

那么怎么判断这个是哪个session的嘞，就靠那个session_id来判断了。当第二次发请求的时候，就会将第一次服务器响应中的session_id放在请求中一起发到服务器中，服务器从请求中提出session_id，和保存的所有session_id比较，然后再找到为这个session开辟的空间。

缺点：

‘Every coin has two sides’,所以嘞，session也有不足危害滴（好像说的有点严重…

因为session是在认证用户，发起请求滴时候，服务器需要去创建一个记录去存储介个，我是理解为给它开一个内存滴。

所以咧，当你的用户足够多的情况下，你的内存可能就会遭不住，因为开销太大辽。
